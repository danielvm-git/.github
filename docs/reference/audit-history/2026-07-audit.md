Full audit, based on the actual local clones in `/Users/danielvm/Developer` rather than the GitHub API (which rate-limited at 60 req/hr unauthenticated). Scanned all 28 repos from the dashboard; 22 have `.github/workflows`, 54 workflow files total. 6 repos have no CI at all: `big-clipboard-manager`, `big-kickass-readme`, `big-server-monitor`, `bigpowers-showcase`, `clean-install-guide`, `skill-method-manager`.

The three "cheap to add, rarely bothered with" items (`permissions:`, `timeout-minutes`, `concurrency`) are the ones that didn't make it into most repos' boilerplate.

## What you're already doing right

This is the more important finding: your workflows are noticeably better than the median repo the best-practices articles are written for.

Least-privilege permissions are set almost everywhere. `pr-review.yaml` scopes to exactly `issues: write, pull-requests: write, contents: read` — that's the textbook example of "don't grant more than the job needs." `docs.yaml` scopes to `pages: write, id-token: write` only. Most teams never bother with this.

Your two highest-risk third-party actions (the ones with write access to `gh-pages` in `ci.yml`) are already pinned to full commit SHAs — `simple-elf/allure-report-action@e463a47...` and `peaceiris/actions-gh-pages@47f197a...`. That's the single most important security control from the official GitHub hardening guide, and you're already applying it where it matters most.

`release-deploy.yml` has a genuinely good deployment pattern: backs up the current binary and database before deploying, health-checks the new version with retries, and automatically rolls back to the previous release if health checks fail. That's more mature than most production deploy workflows.

`big-token-saver/ci.yml` splits fast checks (lint, unit test — run on every push) from a heavy installer verification (12+ tool downloads — `workflow_dispatch` only). That's exactly the "run only what you need" principle. It also pins runner versions (`ubuntu-22.04`, `macos-14`) rather than `-latest`, which none of the other three workflows do.

Secrets are handled correctly — passed through `env:`/`secrets:` blocks, never interpolated directly into `run:` strings.

## The repo that's already doing it right: bigflint

Worth calling out on its own — `bigflint` has 12 workflow files and is the only repo in the portfolio with real SHA-pinning discipline (`publish.yml` and `release.yml` each pin 5 third-party actions to full commit SHAs; `nightly.yml` pins 2). It also has `codeql-analysis.yml` running dependency/vulnerability scanning, which no other repo has. If you're looking for an internal template to copy from rather than writing one from scratch, `bigflint`'s workflow set is the strongest starting point you already own.

## Gaps, ranked by impact

**1. Deploy-time third-party actions aren't SHA-pinned (highest impact).** `release-deploy.yml` uses `appleboy/scp-action@v0.1.7` and `appleboy/ssh-action@v1.2.2` — version tags, not SHAs — to SSH into your Contabo VPS with `CONTABO_SSH_KEY` and push a binary. This is the same class of risk you already mitigated in `ci.yml` (write-access action pinned to SHA), but the deploy workflow is arguably higher-stakes since it holds SSH keys and runs arbitrary commands on a production box. If either maintainer's account were compromised and the tag retagged, that's a direct path to your VPS.

**2. No concurrency guard on deploy.** `release-deploy.yml` triggers on `push: [main]` and `workflow_dispatch`. Nothing stops two deploys from racing if you push and manually dispatch close together — worth adding `concurrency: { group: deploy, cancel-in-progress: false }` (queue, don't cancel, for deploys).

**3. Missing `timeout-minutes` in several places.** `ci.yml`'s `report` job, `pr-review.yaml`, `docs.yaml`, and `big-token-saver/ci.yml` all lack an explicit timeout, meaning if something hangs it can consume the default 6-hour ceiling.

**4. `-latest` runners in 3 of 4 workflows.** `ci.yml`, `pr-review.yaml`, `release-deploy.yml`, and `docs.yaml` all use `ubuntu-latest`. `big-token-saver` shows you already know to pin (`ubuntu-22.04`); worth carrying that habit across repos so a GitHub-side runner image bump can't silently break a build.

**5. No `cancel-in-progress` concurrency on CI (test) jobs.** Only the deploy-adjacent workflows (`docs.yaml`'s `pages` group, `ci.yml`'s `report` job) use concurrency groups. The actual test job in `ci.yml` and `big-token-saver/ci.yml` will happily queue redundant runs if you push multiple commits in quick succession.

**6. Docker-based third-party action pinned to a version tag, not a digest.** `pr-review.yaml` uses `docker://pragent/pr-agent:0.36.0-github_action` — a mutable tag. Since this has `contents`/`pull-requests: write` and calls out to an external API with a secret key, pinning to the image digest (`@sha256:...`) would close the same gap as #1.

## Recommended fix order

1. SHA-pin `appleboy/scp-action` and `appleboy/ssh-action` in `bigbase/release-deploy.yml` — highest real risk in the portfolio.
2. Fix `sqz`'s `dtolnay/rust-toolchain@master`/`@stable` — pin to a version, ideally a SHA.
3. Add `permissions: contents: read` as a top-of-file default to the 28 files missing it (mechanical, scriptable across repos).
4. Add `timeout-minutes` and a `concurrency` group as part of a shared CI template (see learnings doc) rather than repo-by-repo.
5. Consider a reusable `workflow_call` template in a central repo (or via bigbase MCP scaffolding) so these defaults ship automatically in every new project instead of being retrofitted.
